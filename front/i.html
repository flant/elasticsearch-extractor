Background Jobs in GoLang — Your Ultimate Guide to Empower Your Applications

Master efficient background job management in GoLang with Gocraft and Redis.

Modern, sophisticated web development requires leveraging the latest technologies and tools. Gocraft and GoLang offer a potent combination for building robust, scalable, and efficient web applications.

In this blog post, we’ll explore the powerful combination of GoLang, Redis, and Gocraft and how they can take your web development projects to the next level.

Overview

GoLang, a popular programming language known for its speed, concurrency, and efficiency, often requires background processes to handle tasks without blocking the main thread.

Background jobs are tasks executed asynchronously in the background while the main program continues running. This ensures efficient utilization of system resources while avoiding blocking critical functionalities or the user interface.

What are background jobs used for?

Background jobs are typically used for tasks like:

  • Sending emails
  • Processing large datasets
  • Uploading files
  • Scheduled tasks
  • Long-running operations

They offer several benefits:

  • Improved responsiveness: The main program remains responsive while running the background task.
  • Increased scalability: You can quickly scale the number of background workers to handle higher workloads.
  • Better resource utilization: System resources are used more efficiently as the main program doesn’t have to wait for the background task to finish.

Goraft/work is a GoLang library that simplifies the process of running background jobs with Redis. It provides features like:

  • Job queuing: Enqueue and dequeue jobs from a Redis queue.
  • Job arguments: Pass arguments to jobs for customization.
  • Job retries: Automatically retry failed jobs a specified number of times.
  • Job logging: Track the execution status and logs of jobs.
  • Job scheduling: Schedule jobs to run at specific times or intervals.
  • Job concurrency: Control the number of jobs that can run concurrently.
  • Web U.I.: Monitor the job queue and worker performance through a web interface.

Gocraft/work uses a simple architecture with three main components:

  1. Enqueuer: This component adds jobs to the Redis queue.
  2. Worker pool: This is a pool of worker processes that dequeue and execute jobs from the queue.
  3. Web U.I.: This visual interface monitors the job queue and worker performance.

Here’s a basic overview of how Gocraft/work works:

  1. The application enqueues jobs to the Redis queue using the Enqueuer.
  2. Worker processes pick up jobs from the queue and execute them.
  3. The worker processes report their status and logs back to Redis.
  4. The web UI displays the job queue and worker performance information.

Schedulers and Redis Workpools

It would be best to have a scheduler and a workpool to manage background jobs effectively. A scheduler determines when each job should be executed, while a workpool overlooks a pool of worker processes that execute the jobs.

Redis is a popular in-memory data store often used as a backing store for background job queues. It offers high performance and reliability, making it ideal for handling large volumes of jobs. Redis offers several installation options depending on your operating system and preferences. You can choose between precompiled binaries, building from source, or using official package repositories.

A quick overview of the installation process for each platform can be found here — https://redis.io/docs/install/install-redis/

Getting started with Gocraft/work

To get started with Gocraft/work, you can follow these steps:

  1. Install the gocraft/work library using:
go get github.com/gocraft/work

2. Download and install the binary for the Web UI:

go get github.com/gocraft/work/cmd/workwebui
go install github.com/gocraft/work/cmd/workwebui

3. To run the Web UI, you can use the command:

workwebui -redis="redis:6379" -ns="application_namespace" -listen=":5040"
  1. “-redis” — This argument specifies the address and port of the Redis server hosting the job queue.
  2. “-ns” — This argument defines the namespace used to identify the job queue.
  3. “-listen” — This argument specifies the port on which the web U.I. will listen for connections.

Follow a directory structure like this:

example
├── scheduler
│   ├── scheduler.go
├── processors
│   └── process.go
└── go.mod

Enqueue Jobs

Redis Pool:

  1. redisPool: This variable defines a pool of Redis connections. It ensures efficient and reusable connections to the Redis server.
  2. MaxActive: This specifies the maximum number of active connections in the pool (e.g., 5).
  3. MaxIdle: This defines the maximum number of idle connections in the pool (e.g., 5).
  4. Wait: This controls how the pool behaves when all connections are busy.
  5. Dial: This function defines how the pool establishes new connections to the Redis server at the “:6379” address.

Enqueuer:

  1. Enqueuer: This variable defines an enqueuer object responsible for queueing jobs to the Redis server.
  2. “application_namespace”: This specifies the namespace used for identifying jobs belonging to this application.
  3. redisPool: This provides the pool of Redis connections for the enqueuer.
// main file in scheduler package

package main
import (
 "github.com/gomodule/redigo/redis"
 "github.com/gocraft/work"
)
// Make a redis pool
var redisPool = &redis.Pool{
 MaxActive: 5,
 MaxIdle: 5,
 Wait: true,
 Dial: func() (redis.Conn, error) {
  return redis.Dial("tcp", ":6379")
 },
}
// Make an enqueuer with a particular namespace
var enqueuer = work.NewEnqueuer("application_namespace", redisPool)

func main() {
 // Enqueue a job named "send_email" with the specified parameters.
 _, err := enqueuer.Enqueue("send_welcome_email", work.Q{"email_address": "test@example.com", "user_id": 4})
 if err != nil {
  log.Fatal(err)
 }
}
  • The Enqueue method is invoked on the enqueuer object.
  • “send_welcome_email”: This text specifies the name of the job that needs to be added to the queue.
  • work.Q: This map stores the job’s arguments.
  • “email_address”: This argument specifies the email address to send to.
  • “user_id”: This argument holds additional information about the customer.
  • The Enqueue method returns a job I.D. and an error object.
  • The log.Fatal(err) function logs the error message and exits the program if an error occurs while queueing the job.

Process jobs

// main file in processor package

package main

import (
 "github.com/gomodule/redigo/redis"
 "github.com/gocraft/work"
 "os"
 "os/signal"
)
// Make a redis pool
var redisPool = &redis.Pool{
 MaxActive: 5,
 MaxIdle: 5,
 Wait: true,
 Dial: func() (redis.Conn, error) {
  return redis.Dial("tcp", ":6379")
 },
}
type Context struct{
    email string
    userId int64
}
func main() {
 // Make a new pool. Arguments:
 // Context{} is a struct that will be the context for the request.
 // 10 is the max concurrency
 // "application_namespace" is the Redis namespace
 // redisPool is a Redis pool
 pool := work.NewWorkerPool(Context{}, 10, "application_namespace", redisPool)

 // Add middleware that will be executed for each job
 pool.Middleware((*Context).Log)
 pool.Middleware((*Context).FindCustomer)

 // Map the name of jobs to handler functions
 pool.Job("send_welcome_email", (*Context).SendWelcomeEmail)

 // Start processing jobs
 pool.Start()

 // Wait for a signal to quit:
 signalChan := make(chan os.Signal, 1)
 signal.Notify(signalChan, os.Interrupt, os.Kill)
 <-signalChan

 // Stop the pool
 pool.Stop()
}

func (c *Context) Log(job *work.Job, next work.NextMiddlewareFunc) error {
 fmt.Println("Starting job: ", job.Name)
 return next()
}


func (c *Context) FindCustomer(job *work.Job, next work.NextMiddlewareFunc) error {
 // If there's a user_id param, set it in the context for future middleware and handlers to use.
 if _, ok := job.Args["user_id"]; ok {
  c.userId = job.ArgInt64("user_id")
  c.email = job.ArgString("email_address")
  if err := job.ArgError(); err != nil {
   return err
  }
 }
 return next()
}


func (c *Context) SendWelcomeEmail(job *work.Job) error {
 // Extract arguments:
 addr := job.ArgString("email_address")
 if err := job.ArgError(); err != nil {
  return err
 }

 fmt.Println("Sending email")
  // Sending email using net/smtp library
  // (Email should be using gmail domain and password should be created using App password)
 from, password := "xyz@gmail.com", "sthkxlpixjferhfd"
 to := []string{
   addr,
 }

 smtpHost := "smtp.gmail.com"
 smtpPort := "587"
 message := []byte("This is a Welcome email message.")

 // Authentication.
 auth := smtp.PlainAuth("", from, password, smtpHost)

 // Sending email.
 err := smtp.SendMail(smtpHost+":"+smtpPort, auth, from, to, message)
 if err != nil {
  fmt.Println(err)
  return nil
 }

 fmt.Println("Email Sent Successfully!")
 return nil
}

Scheduled jobs

You can schedule jobs to be executed in the future.

enqueuer := work.NewEnqueuer("application_namespace", redisPool)
secondsInTheFuture := 300
_, err := enqueuer.EnqueueIn("send_welcome_email", secondsInTheFuture, work.Q{"email_address": "test@example.com"})
  • secondsInTheFuture: This variable stores the number of seconds to execute the job in the future. In this case, the value is 300 seconds (5 minutes).
  • EnqueueIn: This method is called on the enqueuer object to schedule a job for delayed execution.

Unique jobs

You can enqueue unique jobs so that only one job with a given name/argument exists in the queue at once.

enqueuer := work.NewEnqueuer("application_namespace", redisPool)
job, err := enqueuer.EnqueueUnique("send_welcome_email", work.Q{"email_address": "xyz@gmail.com"}) // job returned
job, err = enqueuer.EnqueueUnique("send_welcome_email", work.Q{"email_address": "xyz@gmail.com"}) // job == nil -- this duplicate job isn't enqueued.
job, err = enqueuer.EnqueueUniqueIn("send_welcome_email", 300, work.Q{"email_address": "abc@gmail.com"}) // job != nil (diff email_address)

Alternatively, you can provide your key for making a job unique. When another job is enqueued with the same key as a job already in the queue, it will simply update the arguments.

enqueuer := work.NewEnqueuer("application_namespace", redisPool)
job, err := enqueuer.EnqueueUniqueByKey("send_welcome_email", work.Q{"email_address": "xyz@gmail.com"}, map[string]interface{}{"my_key": "586"})
job, err = enqueuer.EnqueueUniqueInByKey("send_welcome_email", 120, work.Q{"email_address": "abc@gmail.com"}, map[string]interface{}{"my_key": "586"})

Periodic enqueueing (Cron)

You can periodically enqueue jobs on your Gocraft/work cluster using your worker pool.

pool := work.NewWorkerPool(Context{}, 10, "application_namespace", redisPool)
pool.PeriodicallyEnqueue("0 * * * * *", "send_welcome_email") // This will enqueue a "send_welcome_email" job every minute
pool.Job("send_welcome_email", (*Context).SendWelcomeEmail) // Still need to register a handler for this job separately

Job concurrency

You can control job concurrency using JobOptions{MaxConcurrency: <num>}. Unlike the WorkerPool concurrency, this controls the limit on the number of jobs of that type that can be active at one time within a single Redis instance. This works by putting a precondition on the enqueuing function, meaning a new job will not be scheduled if we are at or over a job's MaxConcurrency limit. A redis key (see redis.go::redisKeyJobsLock) is used as a counting semaphore in order to track job concurrency per job type. The default value is 0, which means "no limit on job concurrency.”

Note: If you want to run jobs “single threaded” then you can set the MaxConcurrency accordingly:

worker_pool.JobWithOptions(jobName, JobOptions{MaxConcurrency: 1}, (*Context).WorkFxn)

Here’s a simplified breakdown of the critical concepts and functionalities of Gocraft/work:

Job Queues:

  • Each job has its queue named after itself.
  • Queues are list-based in Redis.
  • Jobs are enqueued in the respective queue using LPUSH.

Job Priorities:

  • Each queue can have a priority (1–100000).
  • Higher priority queues have higher chances of getting picked by workers.
  • Workers choose queues probabilistically based on their relative preferences.

Job Processing:

  1. The worker fetches a job from the chosen queue.
  2. The job is moved to the “in-progress” queue automatically with the Lua script.
  3. Worker executes the job and increments the job lock.
  4. If successful, the job is removed from the “in-progress” queue.
  5. If unsuccessful, the job is moved to the “retry” or “dead” queue based on the remaining retries.

Job Retries and Scheduled Jobs:

  • “Retry” and “scheduled job” queues are implemented as Redis z-sets.
  • The score is the timestamp for execution.
  • The “requeuer” periodically checks these queues and moves ready jobs to usual queues.

Workers and Worker Pools:

  • Worker pools manage workers and provide the public API.
  • Start/stop pools, attach jobs/middleware, and set concurrency limits.
  • Each pool spins up N worker goroutines based on concurrency settings.
  • Each worker independently gets jobs from Redis, executes them, and loops.

Unique and Periodic Jobs:

  • Unique jobs ensure only one instance with specific arguments exists per queue.
  • Periodic jobs are enqueued based on cron schedules using a “periodic enqueuer.”

Other Components:

  • Reaper: Requeues jobs left in the “in-progress” queue if the worker crashes.
  • Paused Jobs: Jobs in a queue can be paused by setting a Redis key.
  • Dead Jobs: Jobs exceeding retry limits are moved to the dead queue.
  • Terminology: Clarification of terms like “worker,” “queue,” and “job type.”

We explained key concepts like job queues, priorities, retries, and worker pools and explored powerful features such as periodic jobs, unique jobs, and dead letters.
